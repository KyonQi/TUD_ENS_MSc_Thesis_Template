In this chapter, we present the design of our framework that transforms a standard single-device CNN into a set of communication-efficient distributed execution tasks across a cluster of networked MCUs. We first describe the overall coordinator-worker architecture and a two-stage execution pipeline. Then, we detail the offline preprocessing stage, which quantizes the CNN model and compiles it into an execution plan in JSON format (Section~\ref{sec:offline-preprocessing}), and the online inference stage, which carries out the plan over a lightweight application protocol in TCP/IP (Section~\ref{sec:online-inference}). On top of this runtime inference, we develop two core optimization techniques that alleviate the communication and memory overhead of the system: \emph{block-wise local execution} (Section~\ref{sec:block-wise-local-execution}) and \emph{cross-block data reuse} (Section~\ref{sec:cross-block-data-reuse}). Between these two optimizations, we introduce an \emph{overlap-ratio cost model} (Section~\ref{sec:overlap-ratio-cost-model}) that characterizes when block-wise local execution stops paying off, and for cross-block data reuse we analyze the conditions under which it preserves correctness and improves performance (Section~\ref{sec:cross-conditions}).

\label{sec:system-architecture}

As discussed in Section~\ref{chp:system_design}, a typical MCU offers only hundreds of KBs of SRAM (e.g., less than 512~KB) during runtime inference. \textcolor{red}{While the model weights can be stored in Flash memory, the intermediate activations must reside in SRAM at runtime. For CNNs built from IRBs, the largest tensor at runtime is the output of the expanding convolution layer: each block first expands the channel dimension by an expansion ratio $e$ (e.g., $e=6$ in MobileNetV2), applies a depthwise convolution, and then reduces the channel dimension back to the original size (refs).} The output of the expanding convolution layer is therefore $e$ times larger than the input, which can easily exceed the SRAM capacity of a single MCU. This results in a significant memory bottleneck that prevents the execution of large CNNs on resource-constrained MCUs. 

To address this, we implement an end-to-end pipeline for distributed inference across multiple networked MCUs, as shown in Figure~\ref{fig:system-architecture}, comprising two stages:
\begin{itemize}
    \item \textbf{Offline preprocessing stage}: It performs an 8-bit quantization of the model, and extracts the metadata describing the model weights and structural information. The metadata is emitted as JSON files and serialized into the Flash memory of each MCU at compilation time. An \emph{execution plan} is generated that describes the inference strategy and the data flow between the MCUs.
    
    \item \textbf{Online inference stage}: It executes the model on the distributed system across multiple networked MCUs. A single \emph{coordinator} holds the model metadata and the execution plan, partitions the workload to the \emph{workers} at each step, and reassembles the workers' outputs to produce the final inference result. Each \emph{worker} stores its share of the weights and activations, and performs the assigned computation tasks. The coordinator and workers communicate through a lightweight protocol on top of TCP/IP.
\end{itemize}

\section{Offline Preprocessing}
\label{sec:offline-preprocessing}

The offline preprocessing stage employs a trained model for efficient distributed execution on resource-constrained MCUs. It quantizes the model weights and activations to 8-bit integers, extracts the per-layer metadata required at runtime, and compiles a precomputed communication-aware execution plan. The online inference therefore stays simple and makes no global decisions at runtime.


\subsection{Quantization}
Quantization is a significant step in the offline preprocessing stage, as it reduces the model size and memory footprint, enabling efficient execution on resource-constrained MCUs. We employ an 8-bit post-training quantization (PTQ (refs)) with an asymmetric scheme: activations are stored as \verb|uint8|, weights as \verb|int8|, and biases as \verb|int32| with a separate scale per output channel for the weights. For a convolutional layer, the integer accumulator and the requantized output are computed as follows:

\begin{align}
    \mathrm{acc} &= \sum(x_q - z_{\mathrm{in}})(w_q-z_w)+b_{\mathrm{int32}}, \label{eq:quantized-acc}\\
    y_q &= \mathrm{clip}(\mathrm{round}(\mathrm{acc} \cdot m + z_{\mathrm{out}}), 0, 255), \label{eq:quantized-y}\\
    m &= \frac{s_{\mathrm{in}}s_w}{s_{\mathrm{out}}}, \label{eq:quantized-multiplier}
\end{align}
where $(s,z)$ denotes the scale and zero-point of the input, weight, and output tensors respectively. The multiplier $m$ is precomputed and stored per output channel. The quantization parameters are calibrated on a small number of representative samples. 

\textcolor{red}{For every layer of CNNs, the scales, zero-points, and the precomputed multipliers, together with the structural metadata (e.g., layer type, input/output channels, kernel size, residual links between layers, etc.) are extracted and serialized into a JSON file. It is consumed by the coordinator and as C++ headers into each worker's Flash.}

\subsection{Planner}
The planner compiles the quantized model graph into a communication-aware execution plan.
For each block of layers, it decides whether the block can be executed locally by the workers as a single unit or has to fall back to a layer-wise strategy (Section~\ref{sec:block-wise-local-execution}), and whether the block's output feature map can be safely reused by the next block (Section~\ref{sec:cross-block-data-reuse}).
These decisions are made offline and are strictly followed at runtime.

Lines~\ref{ln:offline-start}--\ref{ln:offline-end} of Algorithm~\ref{alg:plan-execute} summarise the offline planning stage.
The planner walks the blocks of $\mathcal{G}$ in order and creates one plan entry $p$ per block.
For each block, it first determines the \emph{execution mode} (line~\ref{ln:block-mode}), which decides whether the block can be executed by the workers as a single locally-computed unit.
It then evaluates whether \emph{cross-block reuse} can be enabled (line~\ref{ln:reuse-enable}), which the system can prevent repeated activations from being transferred across the network.
The completed entry is appended to $\mathcal{P}$, and the resulting plan is what the runtime consumes.

\begin{algorithm}[t]
\small
\caption{The communication-aware execution planning and inference in our proposed framework}
\label{alg:plan-execute}
\KwIn{Model graph $\mathcal{G}$; workers $\mathcal{W}$; input $X$.}
\KwOut{Final prediction $Y$.}

\textbf{Offline planning:}\\
$\mathcal{P}\gets\emptyset$ \label{ln:offline-start}\\
\For{each block $B$ in $\mathcal{G}$}{
    $p\gets\textsc{CreatePlanEntry}(B)$\\
    \eIf{$B$ can be executed locally by workers}{
        $p.\textit{mode}\gets\textsc{BlockWiseLocalExecution}$\label{ln:block-mode}\\
    }{
        $p.\textit{mode}\gets\textsc{LayerWiseFallback}$\\
    }

    \eIf{the output of $B$ can be safely reused by the next block}{
        $p.\textit{reuse}\gets\textsc{EnableCrossBlockReuse}$\label{ln:reuse-enable}\\
    }{
        $p.\textit{reuse}\gets\textsc{DisableCrossBlockReuse}$\\
    }

    $\mathcal{P}\gets\mathcal{P}\cup\{p\}$ \label{ln:offline-end}\\
}

\textbf{Online inference:}\\
$T\gets\textsc{Quantize}(X)$ \label{ln:online-start}\\
\For{each plan entry $p$ in $\mathcal{P}$}{
    \For{each worker $w$ in $\mathcal{W}$}{
        $m_w\gets\textsc{BuildTask}(p,T,w)$\label{ln:build-task}\\
        $\textsc{SendTask}(w,m_w)$\\
    }

    \For{each worker $w$ in $\mathcal{W}$}{
        $r_w\gets\textsc{ReceiveResult}(w)$\label{ln:receive-result}\\
    }

    $T\gets\textsc{CombineResults}(\{r_w\}_{w\in\mathcal{W}})$\label{ln:combine-results}\\
}

$Y\gets\textsc{DecodePrediction}(T)$ \label{ln:online-end}\\
\Return{$Y$}
\end{algorithm}

\section{Online Inference}
\label{sec:online-inference}
The coordinator--worker architecture is designed to execute the distributed inference tasks efficiently.
As illustrated in Figure~\ref{fig:system-architecture}, the coordinator is responsible for task scheduling and data aggregation.
Following the execution plan produced offline, it partitions the input activations, wraps and assigns the computation tasks to the workers, and gathers their results to form the output activations.
Each worker executes its assigned task by loading the model weights from Flash, selecting the suitable computation kernel, and performing the computation on the input activations.
The online part of Algorithm~\ref{alg:plan-execute} (lines~\ref{ln:online-start}--\ref{ln:online-end}) details this runtime workflow: for each plan entry, the coordinator builds a per-worker task (line~\ref{ln:build-task}), dispatches it, collects the partial results (line~\ref{ln:receive-result}), and combines them before advancing to the next entry (line~\ref{ln:combine-results}).



\subsection{Coordinator Design}
\label{sec:coordinator-design}

\begin{figure}[t]
    \center
    \includegraphics[width=\linewidth]{figs/coordinator-components.pdf}
    \caption{Component architecture of the coordinator.
    The inference engine drives execution and holds all inference state, reaching the worker manager and the task distributor through a unified interface.
    The task distributor obtains its block grouping strategy from a pluggable strategy mapper provided by the offline plan, and all components run on a shared asynchronous I/O event loop.}
    % \vspace{-4mm}
    \label{fig:coordinator-components}
\end{figure}

The coordinator is an event-driven server, built on an asynchronous I/O event loop to handle multiple concurrent connections from workers.
As shown in Figure~\ref{fig:coordinator-components}, it is organized into a set of decoupled components:
\begin{itemize}
    \item \textbf{Worker manager}: maintains the connection state of each worker, including the network connections and the message framing;
    \item \textbf{Inference engine}: drives the execution based on the preprocessed execution plan and holds all inference state;
    \item \textbf{Task distributor}: performs the workload splitting and the communication with workers. Tasks are assigned through a pluggable \emph{strategy mapper} that supplies the block grouping decided offline.
\end{itemize}


% The guiding principle is to keep these components independent behind a unified interface that the inference engine uses to drive execution.
The design principle of the coordinator is to abstract the independent components and provide a unified interface for the inference engine to drive the execution.
The engine is agnostic to \emph{how} tasks reach workers, which is the distributor's responsibility, and to \emph{how} layers are grouped into blocks, which is the strategy mapper's responsibility.
This decoupling makes the coordinator flexible: a single engine can support different models, different numbers of workers, and different strategies for block grouping and task distribution without modification.

\paragraph{Worker Manager.}
The worker manager abstracts the network transport from the rest of the system and owns the TCP connection to each worker.
When a worker connects, it first completes a short registration handshake defined at the application layer: the worker announces itself and reports its clock frequency, and the coordinator replies with a unique worker ID.
Each connection is then tracked by a state machine that maintains the connection state and handles message framing.
% Nagle's algorithm is disabled on every connection so that small task descriptors are sent without delay.
Idle workers are held in a queue from which the inference engine draws when dispatching the next round of tasks.
When the manager detects a disconnection from the workers, it immediately shuts down and resets the status of all existing connections, keeping the transport robust and fault-tolerant.

\paragraph{Inference Engine.}
The inference engine is the stateful core of the coordinator and drives the execution of the inference plan.
It maintains the state of the inference process, including the current block or layer being executed, the input and output activations, and the intermediate results.

Figure~\ref{fig:coordinator-workflow} illustrates the coordinator's workflow as the messages exchanged among its components and the workers.
A worker first joins through a registration handshake: it sends a \emph{Register} message to the worker manager, which replies with an \emph{ACK} once the worker has been assigned an ID.
Driven by the engine, the task distributor then consults the \emph{strategy mapper} for the block grouping decided offline and dispatches the per-worker tasks to the worker manager.
The worker manager forwards each task to its worker as a \emph{Task} message and, once the worker finishes, returns the corresponding \emph{Result} message.
The distributor collects these results and issues an \emph{Update Inference} to the engine, which updates the inference state and advances to the next block.
This loop repeats until every block in the plan has been executed. During the inference, the engine also handles error conditions such as worker disconnections or timeouts, retrying or reassigning tasks as needed.

\paragraph{Task Distribution.} 
The task distributor carries out the splitting of workload and the communication with workers. Compared with the \emph{inference engine}, it is stateless with respect to the inference process. Consider a layer (or block) that consumes a feature map of shape $(C,H,W)$, where $C$ is the number of channels, $H$ is the height, and $W$ is the width, and whose convolution has kernel size $k$, stride $s$, and padding $p$. The output height of this layer is
\begin{equation}
    H_{\mathrm{out}} = \left\lfloor \frac{H + 2p - k}{s} \right\rfloor + 1,
\end{equation}
and the output width $W_{\mathrm{out}}$ is computed similarly. 

The distributor splits the $H_{\mathrm{out}}$ output rows into $N$ chunks, where $N$ is the number of workers. Worker $i$ is therefore produces the output rows $[o_s^i, o_e^i)$, where the start and end rows are computed as:
\begin{align}
    o_s^i &= i\left\lceil \frac{ H_{\mathrm{out}}}{N} \right\rceil, \\
    o_e^i &= \min\left((i+1)\left\lceil \frac{ H_{\mathrm{out}}}{N} \right\rceil, H_{\mathrm{out}}\right).
\end{align}

Due to the nature of convolution, the input rows required by worker $i$ have to extend beyond the output row band. For each worker, the range of input rows $[i_s^i, i_e^i)$ is computed as:
\begin{align}
    i_s^i &= \max\left(o_s^i s - p, 0\right), \\
    i_e^i &= \min\left(o_e^i s - p + k, H\right).
\end{align}  

Therefore, two adjacent workers $i$ and $i+1$ have overlapping input rows bounded by $k-s$ rows. It is the redundant computation overhead that will be further discussed in Section~\ref{sec:overlap-ratio-cost-model}. 

For the fully connected layer, the distributor simply splits the the layer by output class with $N$ chunks and assigns worker $i$ to compute the output classes $[o_s^i, o_e^i)$, where
\begin{align}
    o_s^i &= i\left\lceil \frac{ C_{\mathrm{out}}}{N} \right\rceil, \\
    o_e^i &= \min\left((i+1)\left\lceil \frac{ C_{\mathrm{out}}}{N} \right\rceil, C_{\mathrm{out}}\right).
\end{align}

\subsection{Worker Design}
\begin{figure}[t]
    \center
    \includegraphics[width=\linewidth]{figs/worker_state_machine.pdf}
    \caption{Worker state machine, comprising an initialization phase (\emph{Disconnected} through \emph{Idle}) and an event-driven loop that processes one task at a time.}
    % \vspace{-4mm}
    \label{fig:worker-state-machine}
\end{figure}

A worker is a lightweight executor driven entirely by the tasks it receives from the coordinator without a scheduler of its own.
As shown in Figure~\ref{fig:worker-state-machine}, there are two phases during its life cycle.

During the \emph{initialization phase}, the worker moves from \emph{Disconnected} to \emph{Connecting} as it establishes the TCP connection, then to \emph{Registering}, where it completes the registration handshake with the worker manager (Section~\ref{sec:coordinator-design}) by announcing itself and receiving its worker ID in the application layer.
It then settles into \emph{Idle}, ready to serve tasks.

The worker then runs an \emph{event-driven loop}. On receiving a task it enters \emph{Receiving Task}, reconstructs the input activations from the received data, and loads the corresponding weights from Flash.
It transitions to \emph{Computing}, where it selects the appropriate computation kernel based on the layer type and the available memory, and performs the computation.
Finally, in \emph{Sending Result} it returns the output activations to the coordinator and falls back to \emph{Idle} to await the next task.
A malformed or out-of-order task instead takes the \emph{Task Invalid} transition, which is reported to the coordinator rather than executed.

A natural strategy to orchestrate the distributed execution of a CNN is to split the model in a layer-wise manner, where the coordinator splits a layer by rows, dispatches the tasks to workers, collects the partial outputs, reassembles the full feature map, and re-scatters it for the next layer. 
This strategy is simple and straightforward, widely adopted by most distributed inference runtimes~[\textcolor{red}{refs}]. 

However, it incurs a significant communication overhead for IRB-based CNNs as discussed in Section~\ref{sec:IRB}. 
After the expanding layer, the number of channels is expanded by a factor of $e$, resulting in a large feature map that must be communicated between the coordinator and workers. 
This large feature map traverses the network twice and must be held in full at the coordinator, which is a significant bottleneck for resource-constrained MCUs. 


Our Block-wise local execution moves the communication boundary from the layer to the block.
As shown in Figure~\ref{fig:block-wise-local-execution}, instead of exchanging data at every layer, each worker executes an entire IRB locally on its assigned spatial input slice.
The coordinator sends only the block input and receives only the projected output, which is typically $e$ times smaller than the expanded activation.
The expanded activation is materialized in the worker's local buffer and never leaves the device.
For the first IRB of MobileNetV2, this removes up to 1176~KB of intermediate traffic per inference.


\paragraph{Coordinator Memory.}
Beyond the traffic reduction, block-wise local execution also shrinks the coordinator's memory footprint.
Under the layer-wise strategy, the coordinator must hold the entire expanded feature map, which typically dominates its SRAM usage.
Consider one IRB with input shape $(C_{\mathrm{in}}, H_{\mathrm{in}}, W_{\mathrm{in}})$, expansion ratio $e$, and output shape $(C_{\mathrm{out}}, H_{\mathrm{out}}, W_{\mathrm{out}})$.
Its peak coordinator memory under the layer-wise strategy is approximately
\begin{equation}
\mathcal{M}^{\mathrm{layer}}_{\mathrm{coord}}(B) \approx e\, C_{\mathrm{in}} H_{\mathrm{in}} W_{\mathrm{in}} + C_{\mathrm{out}} H_{\mathrm{out}} W_{\mathrm{out}}.
\label{eq:sd-mem-layer}
\end{equation}

Our block-wise local execution removes the former expanded term completely. The coordinator only needs to hold the block input and output activation. The peak memory footprint of the coordinator under block-wise strategy is therefore:
\begin{equation}
\mathcal{M}^{\mathrm{block}}_{\mathrm{coord}}(B) \approx C_{\mathrm{in}} H_{\mathrm{in}} W_{\mathrm{in}} + C_{\mathrm{out}} H_{\mathrm{out}} W_{\mathrm{out}}.
\end{equation}

Since the output activation is typically much smaller than the expanded activation, the block-wise local execution strategy significantly reduces the memory footprint of the coordinator to:
\begin{equation}
\Delta = \frac{\mathcal{M}^{\mathrm{block}}_{\mathrm{coord}}(B)}{\mathcal{M}^{\mathrm{layer}}_{\mathrm{coord}}(B)} \approx \frac{1}{e}.
\end{equation}  

Take the first IRB of MobileNetV2 as an instance, a layer-wise coordinator must hold the block input and the expanded activation, requiring $196$ KB $+$ $1176$ KB = $1372$ KB. With block-wise local execution, the coordinator only handles the block input and the projected output, reducing the footprint to $196$ KB $+$ $73.5$ KB $=$ $269.5$ KB, which is approximately $\frac{1}{e}$ ($e=6$) of the layer-wise strategy.

\paragraph{Overlap-ratio Cost Model.}
\label{sec:overlap-ratio-cost-model}

\begin{figure}[t]
    \center
    \includegraphics[width=\linewidth]{figs/kernel-overlap.pdf}
    \caption{Overlap.}
    % \vspace{-4mm}
    \label{fig:kernel-overlap}
\end{figure}

Block-wise local execution trades communication for computation, as each worker must compute the overlapping input rows required by its neighbors. Figure~\ref{fig:kernel-overlap} illustrates the overlapping input rows for two adjacent workers $i$ and $i+1$. The number of overlapping rows between two adjacent workers is determined by the kernel size $k$ and stride $s$ of the convolution, and is given by:
\begin{equation}
    R_{\mathrm{overlap}} = k - s.
    \label{eq:overlap}
\end{equation}

Under layer-wise strategy, there is no redundant computation for the overlapping rows, since output rows are partitioned and assigned to a single worker only once. 
However, under block-wise local execution, although the output rows of projection layer are still partitioned and assigned to workers only once, each worker must compute the overlapped slice during the local execution of the depthwise convolution as shown in Figure~\ref{fig:kernel-overlap}. We capture this overlapped redundancy with the \emph{overlap ratio}:
\begin{equation}
    \rho = \frac{1}{H}\sum_{i=0}^{N-1}(y_e^i - y_s^i),
    \label{eq:overlap-ratio}
\end{equation}
where $H$ is the height of the input feature map, $N$ is the number of workers, and $[y_s^i, y_e^i)$ is the input row range assigned to worker $i$. 
The overlap ratio $\rho$ represents the fraction of the input feature map that is redundantly computed by multiple workers. 
A higher overlap ratio indicates more redundant computation, which can offset the communication savings achieved by block-wise local execution. 
The ideal $\rho=1$ indicates there is no overlap among the workers, whereas $\rho\to N$ is the worst case where all workers has to contain the entire input feature map.

Combined~(\ref{eq:overlap}) with~(\ref{eq:overlap-ratio}), we can express the overlap ratio as a function of the kernel size $k$, stride $s$, and the number of workers $N$:
\begin{equation}
    \rho(k, s, N) \approx 1 + \frac{(N-1)(k - s)}{H}.
    \label{eq:overlap-ratio-func}
\end{equation}

(\ref{eq:overlap-ratio-func}) demonstrates three key factors that affect the overlap ratio: 
\begin{itemize}
    \item Number of workers $N$: $\rho$ grows with the worer count, indicating that finer partitioning of the input feature map leads to more overlap and redundant computation.
    \item Kernel size $k$ and stride $s$: A larger kernel size or a smaller stride increases the number of overlapping rows due to the nature of the convolution, leading to a higher overlap ratio.
    \item Height of the input feature map $H$: A taller input feature map reduces the overlap ratio, as the overlapping rows become a smaller fraction of the total input.
\end{itemize}

Therefore, the block-wise local execution effect is therefore strongly position-dependent. For an early high-resolution block where $H=112$ with a $3\times3$ depthwise kernel on $N=4$ workers, the overlap ratio is $\rho\approx 1.05$, indicating a nearly free gain. However, for a deep low-resolution block where $H=7$ with a larger $5\times5$ kernel on $N=4$ workers, the overlap ratio is $\rho\approx 2.7$, indicating a significant amount of redundant computation that can easily offset the communication savings.

\paragraph{Adaptive selection.}
This position dependence motivates us a per-block decision on whether to apply block-wise local execution or fall back to layer-wise strategy. (\ref{eq:model-selection}) formalizes our adaptive selection of execution mode for each block $B$ based on the overlap ratio $\rho(B)$ and a manual tunable threshold $\tau$. If the overlap ratio is below the threshold, we apply block-wise local execution; otherwise, we revert to layer-wise strategy. 
\begin{equation}
    mode(B) = \begin{cases}
        \text{block-wise local execution}, & \text{if } \rho(B) \leq \tau, \\
        \text{layer-wise strategy}, & \text{otherwise},
    \end{cases}
    \label{eq:model-selection}
\end{equation}

The threshold can be determined empirically based on the specific model architecture and hardware characteristics. We set $\tau=2$ on a four-worker cluster. A faster network prefers a smaller $\tau$, since communication is cheap and redundant expansion dominates the cost, while a bandwidth-constrained network prefers a larger $\tau$.

A larger kernel size or a smaller stride increases the number of overlapping rows, leading to a higher overlap ratio. Similarly, increasing the number of workers also increases the overlap ratio, as more workers share the same input feature map.

\section{Cross-block Data Reuse}
\label{sec:cross-block-data-reuse}
% To further optimize performance, we have implemented a cross-block data reuse mechanism. This allows data generated in one block to be efficiently utilized in subsequent blocks, reducing redundant computations and improving overall system efficiency.
Block-wise local execution still leaves redundant communication between consecutive blocks. After a worker finishes block $b$, most of its output is exactly the data it needs to begin block $b+1$. A naive strategy returns this data back to the coordinator and immediately sends it back to the same worker again, wasting the limited bandwidth and inflating coordinator memory. To address this, we propose to reuse cross-block data. As illustrated in Figure~\ref{fig:tbd}, it keeps reusable data on the worker and exchanges only the small boundary region the next block requires.

\textcolor{red}{Rephrase needed.} Rather than discarding its output once finished with block $b$, a worker caches the rows it produced for block $b$ in its local buffer. The interior rows of the output need not cross the network, since they are exactly the rows required by the next block $b+1$. The only data that must still move between two consecutive blocks is the thin boundary region that a worker's neighbor requires but does not produce itself.

Concretely, consider worker $i$ needs input rows $[a_i, b_i)$ for block $b+1$, while it already caches rows $[s_i, e_i)$ from block $b$. The cached overlap is the intersection $[\max(a_i, s_i),\min(b_i, e_i))$, so the only rows worker $i$ lacks are the top and bottom boundary rows,
\begin{equation}
    \beta^i_{\mathrm{top}} = \max(0, s_i - a_i),
    \qquad
    \beta^i_{\mathrm{bot}} = \max(0, b_i - e_i),
    \label{eq:boundary}
\end{equation}
which is produced by the neighbouring workers $i-1$ and $i+1$. 

Therefore, on the downlink the coordinator forwards only the boundary rows each worker lacks instead of a full input patch. On the uplink a worker returns only the boundary rows its neighbours will need instead of its full output. The interior rows are never transmitted over the network, which significantly reduces the communication overhead between consecutive blocks. The coordinator now holds only boundaries and acts as a lightweight scheduler rather than a full data aggregator, lowering its peak memory footprint.

Cross-block data reuse is employed only when it does not compromise the accuracy of the inference stage. The enabling conditions are evaluated in the offline preprocessing stage and encoded in the task metadata at runtime, so the workers simply follow the execution plan. There are two situations that make a worker's local data invalid to reuse.

The first is the residual addition. When a block consumes a residual, the coordinator dequantizes the cached residual and the current block output, adds them together, and requantizes the sum in place as
\begin{equation}
    y = clip(round(\frac{(c-z_c)s_c+(r-z_r)s_r}{s_t} + z_t), 0, 255),
\end{equation}
where $(s,s_c,z_c)$ denotes the current block output with its scale and zero-point, and $(r,s_r,z_r)$ denotes the cached residual. This computation changes the cached data in the coordinator, while the worker's cache still holds the pre-residual output. Therefore, the worker's cache is invalid and cannot be reused for the next block. The following block must fall back to a full transfer that replaces the worker's cache with the coordinator's updated data.

The second situation is global average polling together with the fully connected layer that consumes the pooled output. These operations squeeze the spatial dimensions of the feature map into a single value per channel, leaving no rows to reuse.

As a result, we summarize the conditions that enable cross-block data reuse in Table~\ref{tab:reuse-conditions}. In practice, whether cross-block data reuse is enabled is determined by the architecture of the CNN. It is most valid on the early, high-resolution blocks where the boundary savings are also largest.
