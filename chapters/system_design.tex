In this chapter, we present the design of our framework that transforms a standard single-device CNN into a set of communication-efficient distributed execution tasks across a cluster of networked MCUs. We first describe the overall coordinator-worker architecture and a two-stage execution pipeline. Then, we detail the offline preprocessing stage, which quantizes the CNN model and compiles it into an execution plan in JSON format (Section~\ref{sec:offline-preprocessing}), and the online inference stage, which carries out the plan over a lightweight application protocol in TCP/IP (Section~\ref{sec:online-inference}). On top of this runtime inference, we develop two core optimization techniques that alleviate the communication and memory overhead of the system: \emph{block-wise local execution} (Section~\ref{sec:block-wise-local-execution}) and \emph{cross-block data reuse} (Section~\ref{sec:cross-block-data-reuse}). Between these two optimizations, we introduce an \emph{overlap-ratio cost model} (Section~\ref{sec:overlap-ratio-cost-model}) that characterizes when block-wise local execution stops paying off, and for cross-block data reuse we analyze the conditions under which it preserves correctness and improves performance (Section~\ref{sec:cross-conditions}).

\label{sec:system-architecture}

As discussed in Section~\ref{chp:system_design}, a typical MCU offers only hundreds of KBs of SRAM (e.g., less than 512~KB) during runtime inference. \textcolor{red}{While the model weights can be stored in Flash memory, the intermediate activations must reside in SRAM at runtime. For CNNs built from IRBs, the largest tensor at runtime is the output of the expanding convolution layer: each block first expands the channel dimension by an expansion ratio $e$ (e.g., $e=6$ in MobileNetV2), applies a depthwise convolution, and then reduces the channel dimension back to the original size (refs).} The output of the expanding convolution layer is therefore $e$ times larger than the input, which can easily exceed the SRAM capacity of a single MCU. This results in a significant memory bottleneck that prevents the execution of large CNNs on resource-constrained MCUs. 

To address this, we implement an end-to-end pipeline for distributed inference across multiple networked MCUs, as shown in Figure~\ref{fig:system-architecture}, comprising two stages:
\begin{itemize}
    \item \textbf{Offline preprocessing stage}: It performs an 8-bit quantization of the model, and extracts the metadata describing the model weights and structural information. The metadata is emitted as JSON files and serialized into the Flash memory of each MCU at compilation time. An \emph{execution plan} is generated that describes the inference strategy and the data flow between the MCUs.
    
    \item \textbf{Online inference stage}: It executes the model on the distributed system across multiple networked MCUs. A single \emph{coordinator} holds the model metadata and the execution plan, partitions the workload to the \emph{workers} at each step, and reassembles the workers' outputs to produce the final inference result. Each \emph{worker} stores its share of the weights and activations, and performs the assigned computation tasks. The coordinator and workers communicate through a lightweight protocol on top of TCP/IP.
\end{itemize}

\section{Offline Preprocessing}
\label{sec:offline-preprocessing}

The offline preprocessing stage employs a trained model for efficient distributed execution on resource-constrained MCUs. It quantizes the model weights and activations to 8-bit integers, extracts the per-layer metadata required at runtime, and compiles a precomputed communication-aware execution plan. The online inference therefore stays simple and makes no global decisions at runtime.


\subsection{Quantization}
Quantization is a significant step in the offline preprocessing stage, as it reduces the model size and memory footprint, enabling efficient execution on resource-constrained MCUs. We employ an 8-bit post-training quantization (PTQ (refs)) with an asymmetric scheme: activations are stored as \verb|uint8|, weights as \verb|int8|, and biases as \verb|int32| with a separate scale per output channel for the weights. For a convolutional layer, the integer accumulator and the requantized output are computed as follows:

\begin{align}
    \mathrm{acc} &= \sum(x_q - z_{\mathrm{in}})(w_q-z_w)+b_{\mathrm{int32}}, \label{eq:quantized-acc}\\
    y_q &= \mathrm{clip}(\mathrm{round}(\mathrm{acc} \cdot m + z_{\mathrm{out}}), 0, 255), \label{eq:quantized-y}\\
    m &= \frac{s_{\mathrm{in}}s_w}{s_{\mathrm{out}}}, \label{eq:quantized-multiplier}
\end{align}
where $(s,z)$ denotes the scale and zero-point of the input, weight, and output tensors respectively. The multiplier $m$ is precomputed and stored per output channel. The quantization parameters are calibrated on a small number of representative samples. 

\textcolor{red}{For every layer of CNNs, the scales, zero-points, and the precomputed multipliers, together with the structural metadata (e.g., layer type, input/output channels, kernel size, residual links between layers, etc.) are extracted and serialized into a JSON file. It is consumed by the coordinator and as C++ headers into each worker's Flash.}

\subsection{Planner}
The planner compiles the quantized model graph into a communication-aware execution plan. For each block of layers, it decides whether the block can be partially executed locally by the workers as a single unit, or have to fall back to a layer-wise strategy (Section~\ref{sec:block-wise-local-execution}), and whether the block's output feature map could be safely reused by the next block (Section~\ref{sec:cross-block-data-reuse}). These decisions are determined offline and will be strictly followed at runtime. Algorithm~\ref{alg:xx} summarizes the workflow both in the offline preprocessing stage and the online inference stage.

\section{Online Inference}
\label{sec:online-inference}
The coordinator-worker architecture is designed to execute the distributed inference tasks efficiently. As demonstrated in Figure~\ref{fig:tbd}, a coordinator is responsbile for the task scheduling and data aggregation. Based on the execution plan, the coordinator partitions the input activations, wraps and assigns the computation tasks to workers, and gather results from workers to produce the output activations. Each worker is in charge of executing the assigned tasks, which includes loading model weights in Flash, selecting the suitable computation kernel, and performing the computation on the input activations.

\subsection{Coordinator Design}
\textcolor{red}{Add a figure to illustrate the components in coordinator}
The coordinator is an event-driven server, built on an asynchronous I/O event loop to handle multiple concurrent connections from workers. As shown in Figure~\ref{fig:tbd}, it is organized into a set of decoupled components, including:
\begin{itemize}
    \item \textbf{Worker manager}: It maintains the connection state of each worker, containing the network connections and the message framing;
    \item \textbf{Inference engine}: It drives the execution based on the preprocessed plan, abd holds all inference states;
    \item \textbf{Task distributor}: It performs the splitting of workload and the communication with workers. Tasks are assigned to workers with a pluggable \emph{strategy mapper} that supplies the block grouping decided offline.
\end{itemize}

The design principle of the coordinator is to abstract the independent components and provide a unified interface for the inference engine to drive the execution. The inference engine is agnostic to \emph{how} tasks reach workers, which is the distributor's responsibility, and to \emph{how} layers are grouped into blocks, which is the strategy mapper's responsibility. This decoupling design allows for flexibility in the implementation of the coordinator. A single engine could easily support different models on different number of workers, and different strategies for block grouping and task distribution. \textcolor{red}{Maybe need to rephrase it.}

\paragraph{Connection Manager.} 
The worker manager abstracts the network transport from the rest of the system. When a worker tries to connect to the coordinator, it will first go through a short registration handshake defined in the application layer. The worker needs to announce itself and provide its clock frequency, so the coordinator can reply with an assigned unique worker ID. \textcolor{red}{Each connection is tracked by a state machine that handles the connection state and the message framing.} Idle workers are kept in a queue from which the engine draws when dispatching the next round of tasks. \textcolor{red}{Nagle's algorithm is
disabled on each connection so that small task descriptors are sent without delay.}


Connection manager is responsible for establishing and maintaining the TCP connections with workers. It maintains and hadles the connection state of each worker, including the network connections and the message framing. It also provides a unified interface for the inference engine to send and receive messages to/from workers. The connection manager is designed to be robust and fault-tolerant, handling worker disconnections and reconnections gracefully.

\paragraph{Inference Engine.} 
The inference engine is a stateful core of the coordinator, which drives the execution of the inference plan. It maintains the state of the inference process, including the current layer/block being executed, the input/output activations, and the intermediate results. 

\textcolor{red}{Figure~\ref{fig:tbd} illustrates the state machine of the inference engine.}  Upon finishing the TCP handshake with the worker, the engine transitions to the \emph{WAIT} state, waiting for applicatin confirmation from the worker. Once the worker is ready and sends a \emph{READY} message, the engine immediately wrapps and sends the first task to the worker, and transitions to the \emph{IDLE} state, waiting for the \emph{Result} defined in the protocol. Upon receiving the result, the engine updates the inference state and sends the next task to the worker. The engine continues this process until all tasks are completed, and transitions back to the \emph{IDLE} state, waiting for the next inference request. The engine also handles error conditions, such as worker disconnections or timeouts, and can retry or reassign tasks as needed.


\paragraph{Task Distribution.} 
The task distributor carries out the splitting of workload and the communication with workers. Compared with the \emph{inference engine}, it is stateless with respect to the inference process. Consider a layer (or block) that consumes a feature map of shape $(C,H,W)$, where $C$ is the number of channels, $H$ is the height, and $W$ is the width, and whose convolution has kernel size $k$, stride $s$, and padding $p$. The output height of this layer is
\begin{equation}
    H_{\mathrm{out}} = \left\lfloor \frac{H + 2p - k}{s} \right\rfloor + 1,
\end{equation}
and the output width $W_{\mathrm{out}}$ is computed similarly. 

The distributor splits the $H_{\mathrm{out}}$ output rows into $N$ chunks, where $N$ is the number of workers. Worker $i$ is therefore produces the output rows $[o_s^i, o_e^i)$, where the start and end rows are computed as:
\begin{align}
    o_s^i &= i\left\lceil \frac{ H_{\mathrm{out}}}{N} \right\rceil, \\
    o_e^i &= \min\left((i+1)\left\lceil \frac{ H_{\mathrm{out}}}{N} \right\rceil, H_{\mathrm{out}}\right).
\end{align}

Due to the nature of convolution, the input rows required by worker $i$ have to extend beyond the output row band. For each worker, the range of input rows $[i_s^i, i_e^i)$ is computed as:
\begin{align}
    i_s^i &= \max\left(o_s^i s - p, 0\right), \\
    i_e^i &= \min\left(o_e^i s - p + k, H\right).
\end{align}  

\textcolor{red}{Therefore, two adjacent workers $i$ and $i+1$ have overlapping input rows bounded by $k-s$ rows.} It is the redundant computation overhead that will be further discussed in Section~\ref{sec:overlap-ratio-cost-model}. 

For the fully connected layer, the distributor simply splits the the layer by output class with $N$ chunks and assigns worker $i$ to compute the output classes $[o_s^i, o_e^i)$, where
\begin{align}
    o_s^i &= i\left\lceil \frac{ C_{\mathrm{out}}}{N} \right\rceil, \\
    o_e^i &= \min\left((i+1)\left\lceil \frac{ C_{\mathrm{out}}}{N} \right\rceil, C_{\mathrm{out}}\right).
\end{align}

\subsection{Worker Design}
\textcolor{red}{Could draw a workflow fig with the stick one.} A worker is a executor driven entirely by the task it receives from the coordinator. As shown in Figure~\ref{fig:tbd}, upon receiving a task, it
\begin{enumerate}
    \item reconstructs the input activations from the received data, and loads the corresponding weights from Flash;
    \item selects the appropriate computation kernel based on the layer type and available memory resources;
    \item returns the computed output activations to the coordinator, and waits for the next task.
\end{enumerate}

\section{Block-wise Local Execution}
\label{sec:block-wise-local-execution}
% To optimize performance and resource utilization, we have implemented a block-wise local execution strategy. This approach allows us to process data in smaller chunks, reducing memory overhead and improving execution speed. Each block is processed independently, allowing for parallel execution and better load balancing across the system. This design choice also enhances fault tolerance, as failures in one block do not affect the processing of others.

A natural strategy to orchestrate the distributed execution of a CNN is to split the model in a layer-wise manner, where the coordinator splits a layer by rows, dispatches the tasks to workers, collects the partial outputs, reassambles the full feature map, and re-scatters it for the next layer. This strategy is simple and straightforward, widely adopted by most distributed inference runtimes~\cite{tbd,tbd}. However, it incurs a significant communication overhead for IRB-based CNNs as discussed in~\ref{sec:tbd}. After the expanding layer, the number of channels is expanded by a factor of $e$, resulting in a large feature map that have to be communicated between the coordinator and workers. This large feature map traverses the network twice and must be held in full at the coordinator, which is a significant bottleneck for resource-constrained MCUs. 

Our block-wise local execution strategy moves the communication boundary from the layer to the block. As shown in Figure~\ref{fig:tbd}, instead of communicating every layer, each worker executes the entire IRB locally on its assigned spatial input slice. The coordinator only sends the worker block input and receices only projection output, which is typically $e$ times smaller than the expanding output. The expanded activation is materialized locally in the worker buffer and is never transmitted over the network. \textcolor{red}{For the first IRB of MobileNetV2, our block-wise local execution strategy reduces the unnecessary communication overhead up to 1176~KB.}

\paragraph{Coordinator Memory.}
Besides the reduction in communication overhead, block-wise local execution also reduces the memory footprint of the coordinator. Under layer-wise strategy, the coordinator must hold the entire feature map of the expanding layer, which usually dominates the SRAM memory usage of the coordinator.

\textcolor{red}{Need to think if it's calculated correctly.}
Consider one IRB with the input shape $(C_{\mathrm{input}}, H_{\mathrm{input}}, W_{\mathrm{input}})$, an expansion ratio $e$, and the block output shape $(C_{\mathrm{out}}, H_{\mathrm{out}}, W_{\mathrm{out}})$, the peak memory footprint of the coordinator under layer-wise strategy could be calculated as follows:
\begin{equation}
\mathcal{M}^{\mathrm{layer}}_{\mathrm{coord}}(B) \approx e C_{\mathrm{input}} H_{\mathrm{input}} W_{\mathrm{input}} + C_{\mathrm{out}} H_{\mathrm{out}} W_{\mathrm{out}}.
\label{eq:sd-mem-layer}
\end{equation}

Our block-wise local execution removes the former expanded term completely. The coordinator only needs to hold the block input and output activation. The peak memory footprint of the coordinator under block-wise strategy is therefore:
\begin{equation}
\mathcal{M}^{\mathrm{block}}_{\mathrm{coord}}(B) \approx C_{\mathrm{input}} H_{\mathrm{input}} W_{\mathrm{input}} + C_{\mathrm{out}} H_{\mathrm{out}} W_{\mathrm{out}}.
\end{equation}

Since the output activation is typically much smaller than the expanded activation, the block-wise local execution strategy significantly reduces the memory footprint of the coordinator to:
\begin{equation}
\Delta = \frac{\mathcal{M}^{\mathrm{block}}_{\mathrm{coord}}(B)}{\mathcal{M}^{\mathrm{layer}}_{\mathrm{coord}}(B)} \approx \frac{1}{e}.
\end{equation}

\section{Cross-block Data Reuse}
\label{sec:cross-block-data-reuse}
To further optimize performance, we have implemented a cross-block data reuse mechanism. This allows data generated in one block to be efficiently utilized in subsequent blocks, reducing redundant computations and improving overall system efficiency.

\section{Conditions to Enable Cross-block Data Reuse}
\label{sec:cross-conditions}
Some conditions for cross-block data reuse include:
\begin{itemize}
    \item Data must be compatible across blocks.
    \item The data must be stored in a format that allows for easy retrieval and reuse.
    \item The system must ensure that the reused data is up-to-date and valid for the current block's processing requirements.
\end{itemize}
