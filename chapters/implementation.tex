\chapter{Implementation}
\label{chp:implementation}
This chapter describes the implementation of the system design presented in Chapter~\ref{chp:system_design} on real hardware. 
The framework consists of two codebases: a Python codebase that implements the preprocessing toolchains and evaluation scripts, and a real deployment codebase that realizes an asynchrnous Python coordinator with a cluster of Teensy 4.1 workers written in C++. 
We first describe the hardware platform employed in the experiments in Section~\ref{sec:implementation-hardware-platform}. 
Section~\ref{sec:implementation-offline-toolchain} describes the offline toolchains that compile a pretained model into manifests and firmware. 
Following this, Section~\ref{sec:implementation-worker-runtime} and Section~\ref{sec:implementation-communication-protocol} describe the runtime implementations of the coordinator and workers on the MCU. 
We further explains how we validate the numerical correctness of the system in Section~\ref{sec:implementation-validation}. 
Finally, we discuss some engineering details in Section~\ref{sec:implementation-miscellaneous}.


\section{Hardware Platform}
\label{sec:implementation-hardware-platform}

\begin{figure}[t]

    % \vspace{-4mm}
    \label{fig:testbed}

Teensy 4.1 board is employed as the worker in our experiments. It is built with an NXP i.MX RT1062 microcontroller. The device has an Arm Cortex-M7 core running at 600 MHz with a hardware floating-point unit. The board has two 512 KB RAM and 8 MB Flash memory without a memory management unit. Network support is provided by an external Ethernet PHY chip with NativeEthernet library. As shown in Figure~\ref{fig:testbed}, the testbed connects up to six boards to a single TP-Link TL-SG116 gigabit switch. To simplify profiling and data collection, a Linux PC servers as the coordinator, while the worker-side execution runs on MCUs.

The pipeline starts from a pretrained PyTorch model.
Each convolution is fused with its following batch normalization and activation into a single convolution, and the model is quantized to INT8 with post-training quantization (PTQ).
The fused and quantized network is then flattened into an ordered list of layers, each annotated with its shapes, operator parameters, and quantization parameters.
This layer list is the metadata from which all manifests are generated, ensuring that the coordinator and the workers remain synchronized.

\paragraph{Coordinator manifest.}
For the coordinator, the manifest contains a list of layers with their corresponding input and output shapes, as well as the quantization parameters for each layer. 
The coordinator parses this manifest as a starting point to schedule the execution of the model across the workers.

The coordinator is implemented as an asynchronous Python system built on the \verb|asyncio| event loop. Section~\ref{sec:coordinator-design} describes our design of the coordinator. To fully utilize the network bandwidth, the coordinator implements a parallelized I/O interface with \verb|asyncio|. The distributor and collector send and receive data to and from the workers in parallel, so $N$ workers can compute concurrently without waiting for each other. 

To support various execution plan in the coordinator, we abstract and implement a generic \verb|strategy mapper| interface that flattens the layer list into a sequence of execution blocks that the engine drives as shown in Figure~\ref{fig:coordinator-components}. Each model in the framework only needs to define and implement their own \verb|strategy mapper| to support different execution plans. We provide two built-in mappers in the framework for easy usage:
\begin{itemize}
    \item \textbf{Layer}: Each layer is regarded as a single execution unit;
    \item \textbf{Block}: Each IRB is decided to be a single execution unit if the overlap-ratio cost is below the preset threshold, otherwise each layer in the IRB is regarded as a single execution unit.

\end{itemize}

\section{Worker Runtime}
\label{sec:implementation-worker-runtime}



The worker runtime implements the characteristics illustrated in Section~\ref{sec:worker-design}. It is written in C++, built on a baremetal MCU environment without RTOS.

Since the Teensy 4.1 board does not have a memory management unit, we statically allocate a double buffer in the RAM for activations and intermediate results. Static buffers offers deterministic memory addresses, avoiding the overhead of dynamic memory allocation and fragmentation. 

Figure~\ref{fig:memory-layout} illustrates our memory layout. RAM1 and RAM2 holds a 350 KB buffer respectively. The two buffers are used in a ping-pong manner, where one buffer is used to store the input data and the other buffer is used to store the output data. After each execution unit, the buffers are swapped. The weights are stored in Flash as read-only arrays, which are loaded into the SRAM during the execution of each execution unit. This design allows for efficient memory usage and avoids the need for additional memory allocation during runtime. 


% Figure~\ref{fig:tbd} demonstrates the state machine of the worker runtime. It starts with the \verb|IDLE| state after establishing a connection with the coordinator. Then it waits and receives the task message in \verb|RECEIVE| state. After receiving, it transitions to \verb|COMPUTE| state to finish the task. Finally, it sends the result message back to the coordinator in \verb|SEND| state and returns to \verb|IDLE| state. The ping-pong buffer mentioned above is used to keep only one input tensor and one output tensor in the SRAM alive at any time, reducing the unnecessary memory footprint at runtime. 

\begin{figure}[t]
    \centering
    \includegraphics[width=\linewidth]{figs/kernel_tradeoff.pdf}
    \caption{Latency--memory trade-off among convolution kernels for a $3\times3$ convolution. Faster kernels consume more SRAM. The worker selects a kernel at runtime based on the available scratch space.}
    \label{fig:kernel-coompare}
\end{figure}


The worker carries several convolution kernels. As shown in Figure~\ref{fig:kernel-coompare}, the kernel with smaller latency usually takes up more SRAM memory. Therefore, we implement the naive convolution kernel as well as the Im2Col + GeMM optimized kernel for workers to select during runtime. To further reduce the kernel latency, we also utilize the SIMD instructions of the Arm Cortex-M7 core with \verb|arm_dot_prod_q15| in CMSIS-DSP library [refs]. Im2Col + GeMM kernel is the default kernel for workers to use. It utilizes the scratch space from the tail of the output buffer and falls back to the naive kernel when the space is insufficient.


\section{Communication Protocol}
\label{sec:implementation-communication-protocol}

\begin{figure}
    \centering
    \includegraphics[width=\linewidth]{figs/header.pdf}
    \caption{A figure to illustrate the header}
    \label{fig:impl-comm-protocol}
\end{figure}


This section gives the byte-level realisation of the protocol whose message vocabulary and task descriptor were designed in Section~\ref{sec:protocol-design}.
Every message travels as a fixed sixteen-byte header followed by a typed payload, all little-endian.
The header consists of a four-byte magic number that the receiver checks to resynchronise the stream, a one-byte message type, a one-byte worker identifier, a four-byte payload length, and six reserved bytes.
The task descriptor is encoded as one fixed-size record, so the worker parses it with a single structure read and no dynamic allocation.
The result message packs the compute time, the codec time, and the output size as three four-byte integers, followed by the raw output bytes.

\section{Validation}
\label{sec:implementation-validation}
To guarantee the numerical correctness of our distributed system, we implement a a Python simulation system that follows the same execution rule as the real hardware system.

The simulation system takes the identical model manifests and input data as the real system. It simulates and compares the exact array equality of the output tensors during the inference. Any error in the output tensors will immediately surface as a mismatch, making it easy to locate the error in the hardware at runtime.  

\section{Miscellaneous}
\label{sec:implementation-miscellaneous}
\paragraph{TCP transport settings.}
To optimize the network performance, we employ two transport settings that significantly influence the latency of the TCP socket. The first is the Nagle's algorithm, which is a TCP optimization that reduces the number of small packets sent over the network by combining them into larger packets. However, this can introduce additional latency in our system, as it may delay the transmission of small messages. Therefore, we disable Nagle's algorithm in the coordinator to ensure that each message is sent immediately without waiting for additional data to be combined.

The second setting is quick acknowledgment option in workers.
It enables the transport layer to send an immediate ACK rather than a delayed one during the network transmission.
Because it is cleared automatically after every packet, we manually enable it every time a worker receives a message from the coordinator. It prevents the system from stalling a large amount of time waiting for the acknowledgment of the previous packet, significantly reducing the overall latency of the system.

\paragraph{Firmware build environments.}
The whole MCU firmware is built with PlatformIO~\cite{here} with separate environments for development, on-device unit testing, and deployment. The development environment enables debugging and logging, while the on-device unit testing environment runs a suite of tests to validate the correctness of the convolution and linear kernels. The deployment environment allows for the automated script to flash the firmware onto the Teensy 4.1 boards. 


\paragraph{On-device memory measurement.}
In the profiling stage, to report the true footprint, the worker measures its own memory rather than relying on static estimates. The free DTCM available to the stack is the gap between the current stack pointer and the linker symbol at the end of the static data,

\begin{equation}
F_{\mathrm{RAM1}} = \mathrm{SP} - E_{\mathrm{bss}}.
\label{eq:impl-freeram}
\end{equation}

Stack usage is measured with a watermark. Before a layer the worker paints a region below the stack pointer with a known word pattern. As the computation runs, its stack frames overwrite the pattern from the top downward. Afterwards the worker scans the region from the bottom and finds the first overwritten word, which gives the peak stack depth of that layer. Combining the per-layer snapshots, the effective peak demand on RAM1 is
%
\begin{equation}
P_{\mathrm{RAM1}} = \max_{\ell}\,\big( d^{\mathrm{RAM1}}_{\ell} + \sigma_{\ell} \big),
\label{eq:impl-peak}
\end{equation}
%
where $d^{\mathrm{RAM1}}_{\ell}$ is the tensor data resident in RAM1 during layer $\ell$ and $\sigma_{\ell}$ is that layer's measured stack peak. Comparing $P_{\mathrm{RAM1}}$ against the fixed $350$~KB allocation reports how much of the static buffer a given model actually needs. These are the worker memory figures used in the evaluation.
